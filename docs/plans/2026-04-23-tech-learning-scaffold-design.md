# Tech Learning Repo, Scaffold Design

**Date:** 2026-04-23
**Status:** Approved, ready to scaffold

## Goal

A single repo that serves as a shareable, postable knowledge base for tech topics the author researches. Must work three ways simultaneously:

1. Browsable on GitHub (READMEs render nicely)
2. Deployed as a static site (shareable URLs)
3. Portable to dev.to / Medium / LinkedIn (pure markdown, no JSX)

## Decisions

| Decision | Choice | Why |
| --- | --- | --- |
| Shareable surface | All-of-the-above: GitHub repo + static site + portable markdown | User wants flexibility without committing to one channel |
| Content granularity | Hybrid: categorized `topics/` (evergreen) + dated `posts/` (time-based) | Durable knowledge base plus easy TIL/blog-style write-ups |
| Static site generator | Astro + Starlight | Modern, fast, MDX-capable, polished defaults, good for docs + blog |
| Cross-post workflow | Manual copy-paste | YAGNI, defer automation until format is proven |
| Deployment | GitHub Pages via GitHub Actions | Free, zero extra accounts, repo = site |
| Starter content | Templates only (no faux content) | 10-second copy-paste to start new topic/post |
| Subtopic support | Yes, file or folder based, nested naturally via filesystem | User request; Starlight handles arbitrary depth |

## Content Conventions

### Topics

- A **topic** is `topics/<category>/<slug>/index.md`, folder + `index.md`
- A **subtopic** is one of:
  - `topics/<category>/<topic>/<subtopic>.md` (flat, lightweight)
  - `topics/<category>/<topic>/<subtopic>/index.md` (folder, when it has assets or nested children)
- Convention: max 3 levels (category → topic → subtopic) to keep nav sane; not enforced
- Topic `index.md` should link to its subtopics explicitly so it reads well on GitHub too

### Posts

- `posts/YYYY-MM-DD-<slug>.md`, flat, date-prefixed, reverse-chrono by filename
- No categories; use `tags` in frontmatter for grouping

## Frontmatter Schemas

### Topic / Subtopic

```yaml
---
title: RAG Basics
description: What retrieval-augmented generation is and when to use it
category: ai              # required on topic index; optional on subtopics
parent: rag               # optional, for subtopics, enables breadcrumbs
tags: [llm, rag, embeddings]
status: draft             # draft | published
created: 2026-04-23
updated: 2026-04-23
---
```

### Post

```yaml
---
title: What I Learned About RAG This Week
description: One-sentence hook for link previews
date: 2026-04-23
tags: [til, ai]
crosspost: [devto, linkedin]     # informational, manual posting
canonical: https://<username>.github.io/tech-learning/posts/2026-04-23-<slug>/
---
```

Schemas enforced via Zod in `src/content/config.ts`, malformed frontmatter fails the build.

## Astro / Starlight Config

- `site: 'https://<username>.github.io'`
- `base: '/tech-learning'` (project-site deployment)
- Starlight sidebar: two top-level autogenerated groups, "Topics" and "Posts"
- Exclude `**/_template/**` and `**/_template.md` from build
- Shiki theme: `github-dark-dimmed`

## Deployment

- `.github/workflows/deploy.yml`: on push to `main`, build and deploy to GitHub Pages via `actions/deploy-pages@v4`
- One-time manual setup: repo Settings → Pages → Source: "GitHub Actions"

## Authoring Workflow

**New topic:**
```
cp -r src/content/docs/topics/_template src/content/docs/topics/<category>/<slug>
# edit index.md, write
npm run dev   # preview
git commit && git push
```

**New post:**
```
cp src/content/docs/posts/_template.md src/content/docs/posts/$(date +%Y-%m-%d)-<slug>.md
```

**Cross-post checklist** (lives in README):
1. Copy rendered markdown (body only, strip Starlight frontmatter)
2. Paste into dev.to / LinkedIn / Medium
3. Set canonical URL to the site version
4. Adjust platform-specific formatting as needed

## Implementation notes (post-scaffold)

- **Templates moved out of `src/content/docs/`** → now live in top-level `templates/`. Reason: Astro's underscore-prefix ignore rule applies to *files*, not *folders*, so `_template/` was being built into the site. Moving templates outside the content tree is cleaner than fighting the loader. Helper scripts reference `templates/` directly.
- **Starlight 0.32.6 `social` config** uses the object form (`{ github: 'url' }`), not the array-of-objects form that landed in later versions. Update if upgrading.
- **Zod pinned to `^3.24.1`** because Starlight 0.32 expects Zod v3; without the pin, npm resolved Zod v4 and schema parsing crashed at build time.

## Out of Scope (YAGNI)

- Cross-posting automation (dev.to API, RSS sync)
- Custom themes beyond minor CSS hooks
- Comments, analytics, newsletter
- Tag index pages beyond Starlight's built-in search
- Enforcement of the 3-level depth convention

## Package Scripts

- `dev`, Astro dev server
- `build`, production build
- `preview`, preview production build
- `new:topic`, bash helper: `cp -r` template to new topic path
- `new:post`, bash helper: `cp` template with today's date prefix
